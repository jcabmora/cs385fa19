Software Quality Assurance

Lecture 10: Integration Testing

Levels of Testing

  • Unit
  • Integration
  • System
  • User Acceptance (UAT)

Why Integration Testing

  • Unit test cover smallest units of functionality as possible
  • Software is composed of many pieces of software working together, we need to test those interactions
  • Issues that arise when two modules interact:
    • Data types: strings vs. bytes vs. chars, integer vs. float, byte array encodings. This is particularly an issue for non static-typed languages (e.g. ruby, python, javascript)
    • Exception handling: a called module expects the caller to handle exceptions
    • Missing elements in responses where the caller expects a predefined format (e.g XML/JSON/YAML)
    • Changes on the public API of a dependency (e.g. upgrades)
    • Components developed by different individuals.
    • Rest APIs

Integration Testing

  • Testing the interface between software units or modules

Integration Testing

  • The structure of integration tests is very similar to that of unit tests
  • The line between unit tests and integration test and between integration tests and system tests sometimes arre blurry.
  • Try to use real/live depended on components (it is still OK to use Test Doubles e.g. external services that we can't control)
  • Fixtures are more complex/expensive to set up than during unit test
  • Test Harness: Tools to set up fixtures, interact with external dependencies, drive the test execution, and to collect results
