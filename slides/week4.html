Software Quality Assurance

Lecture 4: Unit Testing Introduction

Levels of Testing

  • Unit Testing
  • Integration Testing
  • System Testing
  • Acceptance Testing

Unit Testing

“A unit is the smallest testable piece of software, which may consist of hundreds or even just few lines of source code, and generally represents the result of the work of one or few developers. The unit test cases’ purpose is to ensure that the unit satisfies its functional specification and / or that its implemented structure matches the intended design structure ”
B. Beizer, “Software Testing Techniques”.1990.
... In order to have a clear picture let's see how unit tests differ from the other levels of testing...

Integration Testing

  • Unit tests verify units of functionality independently. In contrast, integration tests verify the interaction of two or more units
  • Integration tests verify the interfaces that the units provide and the coupling between them

System Testing

  • Tests the complete integrated system (software and possibly the hardware)
  • Non-Functional tests are included at this level: e.g.: load, performance, reliability, security tests are part of system testing

Acceptance Testing

  • Tests conducted pre-delivery on a live or staging system
  • Executed by users or end customer
  • Typical examples are beta versions.
... Ok, let's get back to Unit Tests...

Unit Testing

  • They test the smallest unit of functionality possible
  • Functions, or object methods

Who should write the Unit Tests?

The Developers!!

  • Unit test require knowledge of the inner workings of the functionality (they are tipically white-box).
  • Developers usually make the decisions of how logic is partitioned in functional units
  • Developers are in the best position to identify edge / corner cases
  • Developers are the ones that mantain the code and possibly affect the tests with their changes

Why write Unit Tests?

  • Find problems early
  • They provide a living documentation of what the designer intended to build
  • Make collaboration easier since the help other maintainers from inadvertently break functionality.

When should Unit Tests be run?

  • Everytime that developers need to verify their work.
  • When new functionality is ready to be merged into a repository
  • Before builds and compilation
  • Before the execution of other levels of testing

Unit Tests and Test Plans relationship

In order to write Unit tests we should define:

  • Preconditions
  • Execution Steps
  • Expected output
  • Post Conditions

What characteristics should Unit test have?

  • They should fail! (tautology)
  • Fast
  • Simple to run and easy to automate
  • Have proper documentation
  • Should provide meaningful feedback
  • Complete
  • Repeatable
  • Should not be brittle(fragile)
  • Idempotent
  • Should clean after they are completed
