Software Quality Assurance

Lecture 11: Performance Testing

Performance

  • When is an application considered to be performant?
  • Depends on perception!

Key Performance Indicators(KPIs)

  • Response Time
  • Throughput
  • Availability
  • Utilization

Bad Performance

  • Lack of performance considerations in application design
  • Applications are not build with scalability in mind
  • Incorrect Load estimations
  • Lack of automated tests
  • Perfomance Testing is left to last minute (application needs to be stable for performance testing)

Levels of Testing

  • Unit
  • Integration
  • System
  • User Acceptance (UAT)

What we have seen so far are all functional tests.

Performance Testing is a different type of testing

  • Addresses Non-Functional requirements!
  • It is non-deterministic
  • Thresholds are subjective
  • It is also a tool for planning

Why Performance testing

  • How many concurrent users can we support?
  • How long it takes for a page to load?
  • How many servers do we need to support 50000 concurrent users?
  • Does a new front end design impacts page load time?
  • Estabish benchmarks

Performace Tests

We need tools!

  • Capture and recreate user activity
  • Creation of test scenarios to represent different mixes of end-user activity
  • Load Injection
  • Analysis/Statistics

Performance Test Requirements

  • Code freeze: Code needs to pass functional tests first
  • Provision the mix aof servers that will be used during the test
  • Network infrastructure is stable
  • Sizing of databases/assets

Server provisioning

  • Ideally server specs are preserved (otherwise benchmarks are useless>
  • Virtualization and cloud infrastructure facilitate automation and what-if testing

Common Types of Performance tests

  • Pipe-clean test
  • Volume Test
  • Stress Test
  • Soak Test

Pipe-clean tests

  • Executed for a single user case for a period of time/number of iterations
  • Validate tests scripts
  • Stablish baseline for a best-case measurement to be able to determine degradation

Volume Tests

  • Classic performance test
  • Application is loaded to different sets of concurrency levels
  • Evaluate availability, concurrency, throughput and response time
  • Close approximation of real application use
  • Sometimes include simulation of the effects of user interaction (e.g. delays and pauses the interaction with a real human would have)

Stress Test

  • Attempts to cause the application or its components to fail
  • The purpose is to determine the capacity threshold of the SUT
  • The stress tests continues until somethin breaks
  • Useful for capacity / cost planning

Soak test

  • Identify problems that may appear after an extended period of time
  • Detect problems related to: memory leaks, garbage collection, growing databases, unsufficient disk space
  • Requires monitoring infrastructure:
    • CPU utilization
    • Available Memory
    • Paging
    • Free Disk Space
    • Open File Descriptors / Open Ports
    • Network interface: packets in/out
    • Network latency
    • RDBMS specific (slow queries, deadlocks, etc)
  • Correlation of data from the injected load and the monitored parameters is vital to ease problem diagnosis

How are Performance Tests Run

  • Capture Non Functional Requirements
  • Provision Test Environment: Equipment, Tools, External Systems, Accounts, Data/Log Capture
  • Prepare Test Scripts for each Use Case (e.g. set up /teardown)
  • Configure Test Tools
  • Execute the test
  • Analisys / Report
