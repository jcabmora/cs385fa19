Review

  • Phases
  • Fixtures
  • Types of input and output
  • Test Doubles

Review: a good test

  • Able to fail!
  • Fast
  • Simple to run and easy to automate
  • Has proper documentation
  • Provides meaningful feedback
  • Complete
  • Repeatable
  • Not brittle(fragile)
  • Idempotent
  • Cleans after it has completed

Obscure Tests

http://xunitpatterns.com/Obscure%20Test.html
  • It is difficult to understand the test at a glancea
  • Test should serve as a statement of how the SUT should behave
  • Eager Test: A single test verifies too much functionality. See also "Assertion Roulette"
  • Mystery Guest: it should not be necessary to look into external resources to understand why a certain outcome is expected.
  • General fixture: A very large fixture is used among many tests, making it hard to understand what parts of the fixture affect each test

Conditional Test Logic

http://xunitpatterns.com/Conditional%20Test%20Logic.html
  • Test contains code that might not be executed
  • Flexible Test: test code verifies different functionality depending how it is run
  • Conditional Verification Logic: prevent the execution of assertions if the SUT fails to return the right objects

Assertion Roulette

http://xunitpatterns.com/Assertion%20Roulette.html
  • Test contains too many assertions and we can't determine exactly which assertion failed
  • Eager Test: A single test verifies too much functionality
  • Missing Assertion Message: Run the same test logic for multiple different inputs

Erratic Tests

http://xunitpatterns.com/Erratic%20Test.html
  • Tests sometimes pass, sometimes fail
  • Interacting tests Caused by using shared fixtures. Test should not depend on the prior execution of other tests
  • Tear Down failures: Resource leakage
  • Non-deterministic tests: Resource Optimism: Test depend on external resources that can't be controlled
